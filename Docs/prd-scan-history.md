## 1. Introduction / Overview

Add lightweight **scan history persistence** to the scan feature using Room with a **source/repo/usecase** architecture.

This history is distinct from future “saved scanned items”:

- **Scan history**: auto-populated on each successful scan (subject to dedupe rules), minimal metadata, capped at 50 rows.
- **Saved items**: separate future feature with richer data and explicit user save flow.

In v1, history is shown in a **right-side modal drawer** on the main scan screen. The drawer is opened from a **History FAB** at top-right.

## 2. Goals

1. Persist minimal scan history rows automatically after successful scans.
2. Enforce a maximum of 50 persisted history rows.
3. Ignore duplicate scans within 10 seconds.
4. Show history in a right overlay drawer on the scan screen.
5. Allow delete-single and clear-all history actions.
6. Keep scanner and history in the **same scan ViewModel**.
7. Keep implementation layered inside `feature:scan:impl` (`presentation`, `domain`, `data`).

## 3. User Stories

- As a user, I want my recent scans to be remembered automatically so I can quickly review what I scanned.
- As a user, I want a history drawer I can open from the scan screen so I can access recent scans without leaving scanning context.
- As a user, I want to remove one history item or clear all history.
- As a user, when permission is denied, I still want to access history and gallery actions.
- As a developer, I want scan history architecture separated into source/repo/usecase so it remains maintainable.

## 4. Functional Requirements

### 4.1 Data model and persistence

1. Create a scan history table in Room for v1 with these fields:
  - `id` (primary key, autogenerated)
  - `timestampEpochMillis`
  - `rawValue`
  - `codeKind` (`QR` or `BARCODE`)
  - `codeFormat` (specific detected format)
  - `source` (e.g. `CAMERA`, `GALLERY`)
2. History rows are inserted automatically from scan success events.
3. History dedupe rule:
  - If a new scan has the same `rawValue` as an existing recent row within the last 10 seconds, **do not insert**, regardless of format/source.
4. Capacity rule:
  - Keep at most 50 rows persisted.
  - After inserting a new valid row, remove oldest rows beyond the newest 50.
5. Sorting:
  - Return rows in descending timestamp order (newest first).

### 4.2 Layering (inside `feature:scan:impl`)

1. `feature:scan:impl` must be structured into:
  - `presentation`
  - `domain`
  - `data`
2. `data` layer in `feature:scan:impl` must contain:
  - Room local source (DAO-facing source abstraction + implementation)
  - Repository implementation
3. `domain` layer in `feature:scan:impl` must contain use cases:
  - `ObserveScanHistoryUseCase`
  - `RegisterScanHistoryUseCase`
  - `DeleteScanHistoryItemUseCase`
  - `ClearScanHistoryUseCase`
4. `presentation` layer (same scan ViewModel) consumes use cases and exposes UI state.
5. No new public dependency from external modules to `feature:scan:impl`; continue to depend via `feature:scan:api`.

### 4.3 Scan screen behavior and UI updates

1. Replace current scan controls layout with FAB placement:
  - Top-right: **History FAB**
  - Bottom-left: **Flashlight FAB**
  - Bottom-right: **Load from Gallery FAB** (still placeholder/no-op for load behavior)
2. History drawer behavior:
  - Opens from right side as modal overlay.
  - Tapping outside closes it.
  - While open, scanning is paused.
3. History row click behavior in v1:
  - No-op.
4. History visibility and permission behavior:
  - History UI and history FAB are available even if camera permission is denied.
  - Gallery FAB is also visible with permission denied.
5. Scanner-only controls/behavior (camera stream) should still respect permission and existing error states.

### 4.4 ViewModel integration

1. Use the existing scan ViewModel (do not create separate history ViewModel).
2. Extend scan `ViewState` to include:
  - `historyItems: List<ScanHistoryItemUi>`
  - `isHistoryDrawerOpen: Boolean`
3. Add scan interactions for history:
  - `OpenHistoryDrawer`
  - `CloseHistoryDrawer`
  - `DeleteHistoryItem(id)`
  - `ClearHistory`
4. On drawer open:
  - set drawer state open
  - pause scanner state
5. On drawer close:
  - set drawer state closed
  - resume scanner state (when permission and error state allow)
6. Register history row from code-detected interaction using `RegisterScanHistoryUseCase`.

### 4.5 Source field

1. `source` is mandatory on each history row.
2. In v1:
  - camera scans use `CAMERA`
  - gallery placeholder should not create rows yet (until gallery scan implementation exists)

## 5. Non-Goals (Out of Scope)

- Saved scanned items feature (separate domain from history).
- Prompting user to force-save duplicates within dedupe window (future behavior).
- History item details screen.
- History item tap actions (copy/share/open).
- Gallery scan implementation itself (button remains placeholder).
- Backend/cloud sync for history.

## 6. Design Considerations

- Keep existing neobrutal style from `uiComponent` for FABs and drawer content.
- Drawer should feel lightweight and fast; prioritize readability of minimal row data.
- Row UI should at least display:
  - code value (truncated if long),
  - type/format,
  - timestamp,
  - source.

## 7. Technical Considerations

1. Continue using Room for local persistence.
2. Keep scan history data contracts isolated from future “saved item” entities to avoid coupling.
3. Prefer mapping:
  - Entity ↔ Data model in `data`
  - Data model ↔ UI model in `presentation`
4. Dedupe check should be done in repository/local-source transaction-safe logic where possible.
5. Capacity trimming should happen atomically with insertion to avoid temporary overflow.

## 8. Success Metrics

1. New valid scans persist automatically and appear in drawer.
2. Duplicate scans within 10 seconds are not added.
3. Persisted row count never exceeds 50.
4. Delete one and clear all work correctly.
5. History drawer opens/closes correctly; outside tap closes.
6. Scanner pauses while drawer is open and resumes when closed.
7. Behavior works when camera permission is denied (history + gallery FAB still available).
8. Data survives app restart.
