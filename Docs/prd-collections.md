## 1. Introduction / Overview

Replace the current "scan history" data model with a unified **scanned items** model where every scanned code is **permanently persisted** and belongs to exactly one **collection**.

A **collection** is a user-defined box (with a name, a color, and timestamps) that groups related codes — for example "Receipts", "Wi-Fi networks", "Concert tickets". Items that are scanned without an explicit destination land in a special, system-managed **Unspecified** collection that always exists.

This PRD covers:

1. The new persistence model (one entity for items, one entity for collections).
2. The new **Collections** screen (browse + create + open a collection).
3. The new **Collection detail** screen (search items + manage the collection).
4. Changes to the **item details** screen (show + switch collection, edit optional name).
5. Changes to the **Recent / History** screen (display-only window over the unified items table).
6. Changes to the **scan flow** so scans auto-save to Unspecified.

## 2. Goals

1. Persist every successful scan permanently — no automatic deletion.
2. Group scanned items into user-created collections; default to **Unspecified** when none is chosen.
3. Let users browse, create, rename, color, and delete collections (Unspecified excluded).
4. Let users search items inside a collection and switch an item between collections from the item details screen.
5. Replace the old `scan_history` table cleanly (destructive reset — app is still pre-release).
6. Keep the feature module layered (`presentation` / `domain` / `data`) and ViewModels conformant with `Docs/viewmodel-design-guidelines.md`.

## 3. User Stories

1. As a user, I want every code I scan to be saved forever so I never lose a previously scanned item.
2. As a user, I want to group related codes into collections so I can find them again later.
3. As a user, I want a default "Unspecified" bucket so I don't have to choose a collection for every scan.
4. As a user, I want to create a new collection with a name and a color so I can visually distinguish my collections at a glance.
5. As a user, I want to search for an item by its raw value inside a collection so I can find it quickly.
6. As a user, I want to see how many items are in each collection and when it was last updated so I can tell which collections are active.
7. As a user, I want to open an item's details from either the recent list or a collection and see which collection it belongs to.
8. As a user, I want to move an item to a different collection from the item details screen.
9. As a user, I want to give an item an optional friendly name so I can recognize it without reading the raw value.
10. As a user, when I delete a collection I want to choose whether to delete its items or keep them by moving them to Unspecified.

## 4. Functional Requirements

### 4.1 Data model and persistence

1. Replace `ScanHistoryEntity` and the current minimal `ScannedItemEntity` with a single expanded `ScannedItemEntity`:
   - `id: Long` — primary key, autogenerated
   - `collectionId: Long` — foreign key to `CollectionEntity.id`, **never null**
   - `timestampEpochMillis: Long` — when the item was scanned
   - `rawValue: String`
   - `codeKind: String` — `QR` or `BARCODE`
   - `codeFormat: String` — specific format (e.g. `QR_CODE`, `EAN_13`, `PDF_417`)
   - `source: String` — `CAMERA` or `GALLERY`
   - `customName: String?` — optional user-entered display name (defaults to null)

2. Add `CollectionEntity`:
   - `id: Long` — primary key, autogenerated
   - `name: String` — unique (case-insensitive)
   - `colorHex: String` — hex color string (e.g. `#FFAAAA`) chosen from a fixed palette
   - `createdAtEpochMillis: Long`
   - `updatedAtEpochMillis: Long` — equal to `createdAtEpochMillis` initially; updated when an item is **added to** the collection (move-in counts as an add).

3. The **Unspecified** collection is system-managed:
   - Seeded on first database creation with a fixed, well-known `id` (e.g. `1L`) and name `Unspecified`.
   - Cannot be renamed, recolored, or deleted from any UI flow.
   - Always present in the collections list and **pinned at the top** above all user collections.
   - Excluded from collection-name uniqueness checks at edit time (it can't be edited).

4. Item deletion:
   - Allowed only from the **item details screen** and from the existing **Recent/History trash badge**.
   - Collection detail screens are **read-only** for items in v1 (no swipe-delete on collection item lists).

5. Foreign key behavior:
   - `ScannedItemEntity.collectionId` references `CollectionEntity.id`.
   - Deletion semantics are handled explicitly in the repository (see 4.7.5); do not rely on `ON DELETE CASCADE`.

6. No capacity cap. Items are never auto-trimmed.

### 4.2 Layering (cross-module)

1. `feature:collections:impl` must be structured into:
   - `presentation` — Compose UIs, ViewModels, ViewStates, Interactions.
   - `domain` — use cases and pure models.
   - `data` — local data sources, repository implementations, entity ↔ model mappers.

2. `feature:collections:impl` (`data`) owns:
   - The new `ScannedItemDao` (replacing the old one) and the new `CollectionDao`.
   - Repository implementations: `CollectionsRepository`, `ScannedItemsRepository`.

3. `feature:collections:impl` (`domain`) exposes use cases including (at minimum):
   - `ObserveCollectionsUseCase` — emits all collections with counts and `updatedAt`, Unspecified pinned first.
   - `CreateCollectionUseCase`
   - `RenameCollectionUseCase`
   - `RecolorCollectionUseCase`
   - `DeleteCollectionUseCase` (parameter: `cascadeDeleteItems: Boolean`)
   - `ObserveItemsInCollectionUseCase(collectionId, searchQuery)`
   - `ObserveRecentItemsUseCase(limit = 50)` — global, across all collections.
   - `GetItemByIdUseCase(itemId)`
   - `RegisterScannedItemUseCase` — used by the scan feature; defaults `collectionId` to Unspecified.
   - `MoveItemToCollectionUseCase(itemId, targetCollectionId)`
   - `RenameItemUseCase(itemId, customName: String?)`
   - `DeleteItemUseCase(itemId)`

4. Public surface: `feature:collections:api` re-exports composable entry points (`CollectionsScreenEntryPoint`, `CollectionDetailScreenEntryPoint`, `ItemDetailsScreenEntryPoint`) and the use cases needed by other features (notably `RegisterScannedItemUseCase` for `feature:scan:impl`).

5. `feature:scan:impl` removes its own `RegisterScanHistoryUseCase` + history DAO usage and instead calls the shared `RegisterScannedItemUseCase` exposed via `feature:collections:api`.

### 4.3 Collections screen (list of collections)

1. Layout: a vertically scrolling list of collection rows.
2. **Unspecified** is rendered first, always, with a subtle "system" visual hint and no overflow actions.
3. Each row displays:
   - The collection's color as a leading swatch/accent.
   - The collection name.
   - Item count (e.g. `12 items`). Counts above 99 are clamped to `99+`.
   - Relative "last updated" time (e.g. `updated 2h ago`); for empty collections use `created <date>` or similar.
4. User-created collections below Unspecified are sorted **alphabetically by name (case-insensitive, ascending)**.
5. Tapping a row opens the **Collection detail screen** for that collection.
6. A floating action button (or equivalently prominent neo-brutal button) opens the **Create collection bottom sheet** (see 4.8).
7. Empty state: when only Unspecified exists, show a short hint encouraging the user to create their first collection.

### 4.4 Collection detail screen

1. Header shows the collection name, color accent, and item count.
2. Header has a **search input** (text-only filter) that filters items by `customName` (if set) and `rawValue` (case-insensitive substring match).
3. Body shows items in the collection, sorted **newest first by `timestampEpochMillis`**.
4. Tapping an item opens the **Item details screen**.
5. An overflow menu in the header exposes (user-created collections only):
   - **Rename** → opens an edit bottom sheet (name + color).
   - **Delete** → opens the delete dialog described in 4.4.6.
6. Delete collection dialog:
   - Title: `Delete <name>?`
   - Two confirm actions:
     - **Delete items too** — destroys the collection and all items in it.
     - **Move items to Unspecified** — destroys the collection but reassigns its items to Unspecified.
   - Plus **Cancel**.
7. For the **Unspecified** collection, the overflow menu is hidden (no rename/delete).
8. Item list is read-only in v1 (no swipe actions, no multi-select).

### 4.5 Item details screen changes

1. A **single** navigation route is used regardless of where the user came from:
   ```
   NavRoute.ItemDetails(itemId: Long)
   ```
   (Replace the existing `NavRoute.ScanDetails(id: Long)`. Old call sites in scan + history pass the new item id.)

2. The screen must show:
   - The item's collection — name and color swatch — as a tappable chip in the header area.
   - The item's `customName` (if set) and `rawValue`.
   - All existing metadata already shown today (raw value display, code kind, format, source, timestamp, rendered code, copy/share/delete badges).

3. Tapping the collection chip opens the **Collection picker bottom sheet** (see 4.8) to move the item.
4. On successful move:
   - The item's `collectionId` is updated.
   - The target collection's `updatedAtEpochMillis` is set to "now".
   - The previous collection's `updatedAtEpochMillis` is **not** changed by a move-out.
   - Show a confirmation snackbar: `Moved to <Target>` (no undo).

5. Add an **edit name** affordance (e.g. pencil icon next to the name area) that opens a small inline editor or sheet to set/clear `customName`.
   - Clearing the name reverts the display to `rawValue`.
   - The input enforces a soft maximum of **60 characters** (the input does not accept further characters once 60 is reached).

6. The existing trash badge calls `DeleteItemUseCase` and then navigates back (as today).

### 4.6 Recent (history) screen changes

1. The Recent screen continues to exist and is reachable from the same entry points as today's history.
2. It displays a **display-only window** of the **last 50 items globally**, across **all collections** (including Unspecified), sorted newest first by `timestampEpochMillis`.
3. No items are deleted as part of this view — the cap is purely a display limit.
4. Each row should additionally show the item's collection name (small label or color swatch) so users can tell at a glance where an item lives.
5. Tapping a row opens the new `NavRoute.ItemDetails(itemId)`.
6. The existing trash badge on rows remains functional and deletes the item via `DeleteItemUseCase`.
7. The previous 50-item trim logic (in DAO/repository/use cases) is **removed**. Trimming is purely a query `LIMIT 50` on the read side.

### 4.7 Scan integration changes

1. Successful scans (from camera or gallery) call `RegisterScannedItemUseCase`, which:
   - Inserts into `scanned_item` with `collectionId = UNSPECIFIED_COLLECTION_ID`.
   - Updates Unspecified's `updatedAtEpochMillis` to "now" (an add updates).
2. The existing scan **dedupe rule** (no duplicate `rawValue` within 10 seconds) is preserved and now applies across all collections — duplicates inside the dedupe window are not inserted.
3. The post-scan **snackbar** keeps **only** the `Details` action; no `Save to...` action is added in v1.
4. The post-scan snackbar's `Details` action navigates to `NavRoute.ItemDetails(newItemId)`.
5. Existing dependencies on `feature:scan:impl`'s `ScanHistoryDao` / `RegisterScanHistoryUseCase` are removed in favor of the cross-module use case from `feature:collections:api`.

### 4.8 ViewModels (per `Docs/viewmodel-design-guidelines.md`)

Each new screen has its own ViewModel that follows the standard:

1. **CollectionsViewModel** (`feature:collections:impl/presentation/collections/`)
   - Internal flows: `collections`, `isCreateSheetVisible`, `createSheetState` (name + color), `errorState`.
   - `viewState: StateFlow<CollectionsViewState>` via `combine(...) + stateIn(...)`.
   - `CollectionsInteraction`: `OpenCreateSheet`, `DismissCreateSheet`, `UpdateNewName(String)`, `SelectNewColor(String)`, `ConfirmCreate`, `OpenCollection(Long)`.

2. **CollectionDetailViewModel** (`feature:collections:impl/presentation/detail/`)
   - Constructor takes `collectionId: Long`.
   - Internal flows: `collection`, `items`, `searchQuery`, `isEditSheetVisible`, `editSheetState`, `isDeleteDialogVisible`, `errorState`.
   - Items list is built reactively from `searchQuery` + `ObserveItemsInCollectionUseCase(collectionId, searchQuery)`.
   - `CollectionDetailInteraction`: `UpdateSearch(String)`, `OpenEdit`, `DismissEdit`, `UpdateEditedName(String)`, `SelectEditedColor(String)`, `ConfirmEdit`, `OpenDelete`, `DismissDelete`, `ConfirmDelete(cascade: Boolean)`, `OpenItem(Long)`.

3. **ItemDetailsViewModel** (existing scan details VM may be renamed/relocated)
   - Constructor takes `itemId: Long`.
   - Internal flows: `item`, `collection`, `isPickerSheetVisible`, `isNameEditorVisible`, `nameDraft`, `errorState`.
   - `ItemDetailsInteraction`: `OpenCollectionPicker`, `DismissCollectionPicker`, `PickCollection(Long)`, `OpenNameEditor`, `DismissNameEditor`, `UpdateNameDraft(String)`, `ConfirmNameEdit`, `Copy`, `Share`, `Delete`.

4. **CollectionPickerViewModel** (a small dedicated VM that powers the picker bottom sheet)
   - Internal flows: `collections`, `isCreateInlineVisible`, `inlineCreateState`, `errorState`.
   - `CollectionPickerInteraction`: `Pick(Long)`, `OpenInlineCreate`, `DismissInlineCreate`, `UpdateInlineName(String)`, `SelectInlineColor(String)`, `ConfirmInlineCreate`.

5. All four ViewModels follow the guidelines: single public `viewState`, multiple private mutable flows, single `onInteraction(...)`, no business logic in composables, lifecycle via `WhileSubscribed(5.seconds)`.

### 4.9 Create / edit / picker UX

1. **Create collection bottom sheet**:
   - Fields: text input for `name`, color picker (preset palette of ~6–8 neo-brutal-friendly colors).
   - Buttons: `Cancel`, `Create`. `Create` is disabled until the name is non-empty.
   - On duplicate-name error, show an inline error under the name field; do not dismiss the sheet.

2. **Edit collection bottom sheet**:
   - Same shape as create. Buttons: `Cancel`, `Save`.
   - Hidden entirely for Unspecified (entry point is hidden anyway).

3. **Collection picker bottom sheet** (used from item details "switch collection"):
   - Lists all collections, Unspecified pinned at top, others sorted **alphabetically by name (case-insensitive, ascending)**.
   - First row is `+ New collection` which expands the sheet into an inline create form (name + color).
   - Confirming inline create creates the collection AND moves the item into it in one operation.

### 4.10 Migration

1. The database schema upgrade is **destructive**: drop `scan_history` and the old minimal `scanned_item` table; create the new `scanned_item` and `collection` tables fresh. No user data preservation is required because the app is pre-release.
2. Bump `ScanAppDatabase` version accordingly; rely on `fallbackToDestructiveMigration(true)` (already enabled in `getRoomDatabase`).
3. Seed the **Unspecified** collection during `RoomDatabase.Callback.onCreate` or first repository access (whichever is cleanest); ensure it always uses the well-known id.

## 5. Non-Goals (Out of Scope)

1. Filters or sorts inside a collection beyond text search (v2).
2. Bulk operations (multi-select move/delete) (v2).
3. Drag-and-drop reordering of collections or items (v2).
4. Sharing or exporting an entire collection (v2).
5. Emoji/icon customization for collections (v2; only color in v1).
6. Cloud sync, backup, or cross-device support.
7. Tagging / multi-collection membership (item belongs to exactly one collection).
8. Renaming or deleting the Unspecified collection.
9. Swipe-to-delete or any item mutation from collection detail item lists (read-only in v1).
10. A "Save to..." action on the post-scan snackbar.
11. Undo for move/save operations.

## 6. Design Considerations

1. Continue using the neo-brutal style (`uiComponent` — pastel accents, black outlines, offset shadows).
2. Color palette for collections should match the existing pastel tokens already in `uiComponent.theme` (e.g. `PastelSalmon`, `PastelBlue`, etc.). Add new pastels if more variety is needed.
3. Unspecified should look slightly muted (e.g. neutral gray accent) so users immediately understand it's the default bucket.
4. The collection chip on the item details screen should be tappable and visually clearly a button (border + shadow, neo-brutal style), not a passive label.
5. Bottom sheets should follow whatever bottom-sheet pattern already exists in the codebase (e.g. `ScanMenuModal`); do not invent a new one.
6. The relative time formatting (`2h ago`, `just now`, etc.) should reuse `core:utils` time helpers.

## 7. Technical Considerations

1. Follow `Docs/viewmodel-design-guidelines.md` strictly — single `viewState`, multiple private mutable flows, single `onInteraction`, lifecycle via `WhileSubscribed(5.seconds)`.
2. Mirror the layering already used in `feature:scan:impl` (`presentation` / `domain` / `data`).
3. Reactive item count + last-updated on the collections list should be computed via a single Room query joining `collection` and `scanned_item` (e.g. `LEFT JOIN ... GROUP BY collection.id`), not assembled in Kotlin.
4. Search inside a collection should be done with a `LIKE` query in Room, debounced in the ViewModel (300 ms) before reaching the repository.
5. `RegisterScannedItemUseCase` belongs in the collections domain because items are the collections feature's owned concept; expose it via `feature:collections:api` for consumption by `feature:scan:impl`. Avoid a reverse dependency.
6. Name uniqueness:
   - DB-level: unique index on `collection.name COLLATE NOCASE`.
   - Repository: surface a typed `CollectionAlreadyExists` error to the ViewModel for inline UI handling.
7. The Unspecified seed must run **before** the first scan can complete; safest place is a `RoomDatabase.Callback.onCreate` insert plus a defensive "ensure exists" call at repository construction time.
8. Removing the old `ScanHistoryDao` / `RegisterScanHistoryUseCase` should also remove all dedupe/trim queries; dedupe logic moves into `RegisterScannedItemUseCase` (same 10-second rule applies across all collections).
9. The navigation route rename (`ScanDetails` → `ItemDetails`) must be applied across all `NavHost` cases, scan VM emissions, history screen click handlers, and the post-scan snackbar action.
10. Item `customName` editing should be debounced and persisted on confirm, not on every keystroke.

## 8. Success Metrics

1. Every successful scan results in exactly one new row in `scanned_item` (subject to the 10-second dedupe rule), assigned to Unspecified by default.
2. No automatic deletion of items occurs; row count grows over time (verified by repeated scans across an app restart).
3. The collections list always shows Unspecified pinned at the top, with the correct item count (clamped to `99+` above 99) and "last updated" time; user collections below it are sorted alphabetically (case-insensitive).
4. Creating a collection with a duplicate name shows an inline error and does not create a row.
5. Renaming a collection updates `updatedAtEpochMillis`? — **No**, only adds do (see 4.1.2). Renaming changes `name`/`colorHex` only. Verify in tests.
6. Deleting a collection with the "Move to Unspecified" option moves all items and removes the collection; with "Delete items too" it removes both.
7. Searching inside a collection narrows the list in <300 ms after typing stops.
8. The item details screen opens correctly from both the Recent list and a Collection detail list, and shows the current collection chip.
9. Moving an item updates its `collectionId`, bumps the target collection's `updatedAtEpochMillis`, and shows the `Moved to <Target>` snackbar.
10. The old `scan_history` table no longer exists in the schema after migration; the app starts cleanly on a wiped install.
11. All four new/changed ViewModels conform to the guidelines (single `viewState`, private mutable flows, single `onInteraction`).

## 9. Open Questions

All v1 questions are resolved:

1. **Collection ordering** — Alphabetical by name (case-insensitive, ascending), with Unspecified pinned at the top. Applies to both the collections list (4.3) and the collection picker (4.9.3).
2. **Color palette** — Reuse the existing pastel tokens from `uiComponent.theme` (`PastelSalmon`, `PastelBlue`, etc.). The picker exposes the current palette; new pastels can be added later without a schema change.
3. **Item count rendering** — Show the exact count up to 99; counts of 100 or more are rendered as `99+`.
4. **`customName` length cap** — Soft maximum of **60 characters**, enforced in the input (input refuses further characters once the cap is reached).
5. **History label** — Keep the label "History" in v1.
